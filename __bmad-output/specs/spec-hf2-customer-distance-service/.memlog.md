- (note) Source: HF2_SPEC.md (repo root), fully absorbed — a harness-comparison benchmark spec for a small offline REST service over Postgres.
- (decision) Why = mandate: this is a fixed benchmark task used to compare agentic development harnesses/workflows against the same requirements, not a product solving an end-user pain.
- (capability) CAP-1 intent: system loads seed-customers.json into a customers table and geocodes each customer's city via the bundled reference/city-coordinates.json, re-running without duplicating rows. success: running the seed twice yields the same row count as running it once; every row's lat/lon is populated when its (city, countryCode) matches the reference (case/accent/whitespace-insensitive) or is null with a logged warning when it doesn't, and the seed never crashes on a miss.
- (capability) CAP-2 intent: a client can call GET /customers/count to get the total seeded customer count. success: response body is exactly {"count": N} where N equals the live row count in the customers table.
- (capability) CAP-3 intent: a client can call GET /customers/by-distance to get all customers ordered by ascending distance to Budapest, each annotated with distanceKm. success: response is a list where every element has distanceKm rounded to 1 decimal; order is ascending by distanceKm; Budapest customers show 0 and sort first; customers with unknown coordinates sort last with distanceKm: null; ties are broken by name ascending.
- (constraint) Offline only: no external geocoding API call and no LLM call at runtime, for seeding or serving.
- (constraint) customers table minimum columns: id, telepules, lat (nullable), lon (nullable); budget and note may also be stored but are not required.
- (constraint) City-name matching for geocoding must be accent-insensitive, case-insensitive, and whitespace-trimmed; Budapest (optionally its districts) resolves to the capital's coordinates.
- (constraint) A city absent from the reference file is not an error: store lat/lon as null, log it, and continue seeding the rest.
- (constraint) Haversine distance calculation needs dedicated unit tests covering: a known real distance (Budapest-Vienna, approx. 214 km), the 0 km self-distance case, and null-coordinate handling.
- (constraint) Delivery must land as small, focused commits so the build process itself is inspectable.
- (constraint) A README must cover: starting Postgres, running the migration, seeding, starting the server, and running the tests.
- (non-goal) No authentication/authorization on any endpoint.
- (non-goal) No write endpoints beyond the seed process itself (no create/update/delete customer API).
- (non-goal) No frontend/UI.
- (non-goal) No external geocoding API integration.
- (non-goal) No LLM calls at runtime.
- (assumption) Postgres MCP wiring for development-time schema/data visibility is optional per the source spec (if a tool for it is available) and is not a required deliverable; omitted unless the environment already provides it.
- (decision) Success signal: a single documented command sequence (start Postgres, migrate, seed, start server) yields a running service where GET /customers/count returns 15 and GET /customers/by-distance returns all 15 customers correctly ordered/annotated, demonstrable via curl, plus the three required haversine unit tests passing.
- (assumption) Assumed the customers table stores a 'name' column despite the source's minimum-column list omitting it, because CAP-3's by-distance tie-break explicitly requires sorting by name — the two parts of the source are inconsistent on this point and name-storage is the only reading that satisfies both.
