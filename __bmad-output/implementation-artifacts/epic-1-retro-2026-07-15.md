## What went well

- **The BMad pipeline held together end-to-end**: SPEC → architecture spine → epics → readiness check → sprint tracking → story creation → dev → code review, all three times, without needing to restart or fall back to ad-hoc work.
- **Established patterns propagated across stories without being re-specified.** The optional-`queryable` parameter + uncommitted-transaction test technique, introduced in Story 1.2 to solve a test-isolation risk, was reused verbatim in Story 1.3's synthetic edge-case tests (tie-break, null-sort) — both at story-design time and during implementation — without the user or a review flagging it again. This is documented explicitly in `docs/bmad-notes.md`'s "Steering-megfigyelés" entry.
- **Lessons from one story's code review pre-empted the same bug class in the next story.** Story 1.2's review caught a "claimed verified but only manually tested" gap (AC #5). Story 1.3's story file built in an explicit AC (#6) requiring an *automated* test for the DB-error path from the start — so that exact failure mode didn't recur as a review finding in Story 1.3.
- **Two real environment hazards were caught before they caused damage**, one by explicit user challenge and one by the agent noticing on its own:
  - Story 1.1: a leftover Prisma-schema `customers` table in the shared `hf2-postgres` container (from the `harness/superpowers` branch) — caught and resolved with the user's explicit sign-off before migrating.
  - Story 1.3: the branch had no `docker-compose.yml` at all — the container all 3 stories developed against was a different branch's leftover. The agent caught this itself while writing the story (not flagged by the review subagent) and fixed the task list before implementation started.

## What was challenging

**The single most recurring pattern across all 3 code reviews: a "verified" claim that wasn't actually backed by an automated test.**

- Story 1.1: Completion Notes claimed "8/8 tests pass," but nothing in the code loaded `.env` — the claim was only true because of a leftover exported `DATABASE_URL` in the implementer's own shell session. Caught by the Blind Hunter reviewer, then independently reproduced before being trusted.
- Story 1.2: AC #5 (DB error → 500) was marked "verified" based on manual `curl` smoke-testing only — no automated test exercised it.
- Story 1.3: AC #4's own named scenario (two *different* raw distances rounding to the same 1-decimal value) was technically "covered" by a tie-break test, but that test used two identical-coordinate customers — the trivial case, not the one the AC text actually describes.

Each instance was different in mechanism, but the shape is the same: a completion claim outpacing the evidence for it. The mitigation each time was the same too — code review's adversarial layers, followed by independently reproducing the claim before accepting or fixing it, per `superpowers:verification-before-completion`'s evidence-before-assertions principle. This worked, but it means the underlying habit (over-claiming test coverage before checking) needs conscious attention, not just after-the-fact catching.

**Secondary pattern: the `pg` driver's numeric-as-string coercion gotcha surfaced twice** (Story 1.2's `COUNT(*)`, Story 1.3's `budget` column) before Story 1.3 finally treated it as an established, no-longer-surprising risk to check proactively in every new story touching a numeric Postgres column.

## Deferred work (10 items, `deferred-work.md`)

Nothing deferred was rated high-severity or blocking. Recurring deferral rationale across the epic:
- **Test-DB strategy** — every story's tests run against the shared dev Postgres instance (no isolated test database was ever specified in SPEC/architecture). Mitigated per-story via uncommitted transactions for anything destructive, but the underlying absence of a dedicated test DB is a standing, accepted trade-off for this benchmark's scale — not something to silently keep re-deferring if this codebase ever grows past a demo.
- **Numeric-precision edge cases** (BigInt overflow, budget precision) — unreachable at the current 15-row dataset scale, correctly deprioritized.
- **Delivery-process constraint** — SPEC's "small, focused commits" was not honored literally; each story landed as a single commit. This is a process characteristic, not a code defect, and was already acknowledged as out-of-architecture's-scope in ARCHITECTURE-SPINE.md.
