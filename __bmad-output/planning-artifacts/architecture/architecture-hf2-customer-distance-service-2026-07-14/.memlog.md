- (decision by user) Stack: Node.js + Express + pg (raw SQL, no ORM). Binds: all. Prevents: two units picking incompatible query layers (ORM vs raw) or incompatible migration tooling. Chosen over Python/FastAPI and Go for lowest ceremony given 3 endpoints + 1 seed script and the 'small inspectable commits' constraint.
- (decision by user) Process boundary: seed/geocode runs as a separate CLI script with its own entrypoint (e.g. 'npm run seed'), never invoked by the server on boot. Binds: CAP-1 vs CAP-2/CAP-3. Prevents: server startup time coupling to seed idempotency checks, and seed logic leaking into request-handling code paths.
- (version) Verified 2026-07-14 via web: Node.js 24 (current LTS), Express 5.2.1 (5.x now the TC-recommended active line; 4.x in maintenance since 2025-04, targeted EOL not before 2026-10), pg 8.22.0, node-pg-migrate 8.0.4 (stable; 9.0.0-alpha exists but not adopted).
- (decision by user) Migrations: node-pg-migrate. Binds: CAP-1 (schema). Prevents: schema drift between environments and ad-hoc/undocumented DDL changes; one migrations table tracks applied state.
- (decision by user) Test runner: Node built-in node:test (node --test), no added dependency. Binds: haversine unit tests constraint. Prevents: pulling in a test framework dependency for 3 unit tests, inconsistent with the minimal-dependency stack stance.
- (decision by user) Idempotency mechanism: DB-level UNIQUE constraint on (name, telepules) natural key + INSERT ... ON CONFLICT DO NOTHING. Binds: CAP-1. Prevents: duplicate rows on re-run being only as safe as application-side check logic; pushes the invariant into the schema itself.
- (decision) Paradigm: simple layered (routes -> services/domain -> data access), no hexagonal/ports-and-adapters -- overkill for 3 endpoints and 1 seed script. Binds: all. Prevents: routes reaching directly into pg client calls in some files and through a service layer in others.
- (decision) Haversine distance calculation must live in its own pure, dependency-free module (no DB/HTTP imports) so the 3 required unit tests exercise it directly. Binds: CAP-3, haversine test constraint. Prevents: the calculation being inlined into the route handler where it can't be unit-tested in isolation.
- (decision) DB connection config: single DATABASE_URL env var read in one config/db module, reused by both the seed script and the server. Binds: CAP-1/2/3. Prevents: seed and server each inventing their own env var names or connection-string assembly.
- (decision) City-name normalization (accent/case/whitespace-insensitive matching, Budapest+districts special-case) lives in one exported function used only by the seed/geocode step. Binds: CAP-1. Prevents: two slightly different normalization implementations if geocoding logic is ever split across files.
